# GitHub学習計画（仕組み理解 + AIエージェント活用）

最終更新日: 2026-03-02

## 学習の前提
- 目的は「操作を覚える」だけでなく、「なぜその手順なのか」を説明できる状態になること
- 今後はAIエージェント活用を前提に、`人間が設計責任を持つ` 進め方を身につける
- すべてのセッションで、最後に「自分の言葉で説明」を実施する

## 到達目標
### A. Git/GitHubの基礎運用
- Git/GitHubの基本操作を自力で回せる
- `main` へ直接作業せず、`feature/*` ブランチ運用を習慣化する
- PR作成からマージ、同期、ブランチ削除までを一連で行える

### B. 仕組みの理解
- 変更が `working tree -> staging area -> commit -> remote` で管理される理由を説明できる
- `rebase` と `merge` の違いを履歴の観点で説明できる
- コンフリクト発生時に「何が衝突したか」を説明して解消できる

### C. AIエージェント活用
- AIに依頼する前に、目的・制約・受け入れ条件を言語化できる
- AIが出した提案を鵜呑みにせず、差分・挙動・リスクをレビューできる
- AI活用時も、最終的な設計判断と責任は人間側にあると理解して実践できる

## 学習ロードマップ
### Phase 1: 基本反復 + 説明力（2-3回）
- 同じ流れを別ファイルで繰り返す
- コマンドを見ずに実行できることを目標にする
- 各手順で「なぜ必要か」を1行で説明する

### Phase 2: 実務の基本
- `.gitignore` を理解して設定する
- コミットメッセージ規約（例: `feat:`, `fix:`, `docs:`）を使う
- `git log --oneline --graph --decorate` を読めるようにする
- PRテンプレートに「目的・変更点・確認項目」を書く

### Phase 3: トラブル対応
- コンフリクト解消
- 誤って `add` した変更の取り消し
- 誤ったコミットメッセージの修正（`--amend`）
- 失敗時に「原因・影響・再発防止」を `session-log.md` に記録する

### Phase 4: AIエージェント協働（新設）
- AIにタスク依頼する前に、以下を必ず定義する
  - ゴール（何を達成したいか）
  - 制約（変更不可ファイル、命名規約、期限）
  - 受け入れ条件（テスト、期待挙動、レビュー観点）
- AIの提案に対して、以下を必ず実施する
  - 差分確認（不要変更がないか）
  - 実行確認（コマンド/テスト結果が妥当か）
  - 設計確認（将来の保守・拡張に耐えるか）

## 1セッションの標準フロー
1. `learning-plan.md` の「次にやること」を確認
2. 作業前に「ゴール・制約・受け入れ条件」を3行で書く
3. 実装/操作を実施（必要ならAIを利用）
4. PR前に `git status`, `git diff --staged`, `git log -n 5 --oneline` を確認
5. `session-log.md` に「できたこと・詰まったこと・次回改善」を追記

## 小規模開発と大規模開発の違い（学習観点）
- 小規模: 速度重視。最低限の運用ルール（ブランチ、PR、命名）を守る
- 大規模: 再現性重視。レビュー基準、テンプレート、運用ルールを文書化して統一する

## MVP段階とスケール段階の違い（学習観点）
- MVP段階: まず動くものを短サイクルで作り、学習速度を優先
- スケール段階: 変更頻度の増加に備え、テスト・レビュー基準・監視観点を先に整える

## 次にやること（次回）
1. `feature/practice-ai-1` ブランチを作る
2. `.gitignore` を追加または改善し、理由を1-2行で説明できる状態にする
3. AIに依頼する前に「ゴール・制約・受け入れ条件」を `session-log.md` に下書きする
4. PR作成〜マージ〜同期までを実施する
5. `session-log.md` に以下を追記する
   - 自力でできたこと
   - AIに任せたこと
   - AI提案を修正した点（理由つき）
